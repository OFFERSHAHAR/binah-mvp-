מה אנחנו בונים

סוכן קטן, אבל עם גבולות ברורים

הסוכן לא אמור להיות “כל יודע”. הוא עובד כמו עובד תפעול ממושמע: קורא מקור מידע מוגדר, בודק מה יש בו, מודה כשאין מידע, ולא משלים פערים מהדמיון.

הקלט

שאלה של המשתמש, למשל: “מה המחיר של חבילת Pro?”

מקור האמת

Google Sheet או CSV מקומי עם מוצרים, מחירים, סטטוסים, שאלות ותשובות.

הפלט

תשובה בעברית שמבוססת רק על השורות שנמצאו בטבלה.

הכלל הקדוש: אם התשובה לא נמצאת בנתונים שקראנו, הסוכן אומר שאין לו מספיק מידע.

מסלול מתחילים

איך ניגשים לזה אם אין לך רקע טכני

לא מתחילים מ־AI. מתחילים ממערכת קטנה וברורה: טבלה, תיקייה, כמה קבצי הוראות, סקריפט שמחפש בטבלה, ורק בסוף מודל שמנסח תשובה.

מה אתה צריך להבין

הסוכן לא “יודע”. הוא קורא שורות מטבלה, ואז מנסח תשובה לפי השורות האלה בלבד.

מה אתה לא חייב להבין בהתחלה

לא חייב לדעת Machine Learning, אימון מודלים, שרתים, Docker, או ארכיטקטורות מורכבות.

המטרה של המסלול הזה: להגיע להרצה ראשונה עובדת, גם אם אף פעם לא בנית סוכן.

LLM + Markdown + Scripts

החלוקה שמונעת בלגן

רכיב תפקיד מה לא עושים איתו
LLM מבין את השאלה, בוחר פעולה, מנסח תשובה. לא נותנים לו להמציא נתונים עסקיים.
Markdown מגדיר תפקיד, כללים, תהליך, טון, גבולות. לא משתמשים בו כדי לבצע חישובים או קריאות API.
Scripts קוראים Sheets, מסננים נתונים, בודקים תקינות, מחזירים JSON. לא נותנים להם להחליט איך לדבר עם המשתמש. 
User Question
  -> load Markdown instructions
  -> call Script: read_sheet / search_rows
  -> send data snippet to LLM
  -> LLM answers only from snippet
  -> log question, rows used, final answer

שלב 3

כותבים את קובץ הזהות של הסוכן

`markdown/agent.md` הוא חוזה ההתנהגות. הוא לא אמור להיות פיוטי. הוא צריך להיות חד, קצר, ובלתי ניתן לפרשנות יצירתית מדי. 

# Sheet Answer Agent

## Role
You answer user questions using only rows returned from the sheet search script.

## Hard Rules
- Never answer from general knowledge.
- Never invent prices, dates, availability, policy, or product details.
- If no relevant rows are returned, say that the information was not found in the sheet.
- Mention uncertainty when the matching rows are partial.
- Keep the answer concise and useful.

## Allowed Source
The only source of factual information is the JSON returned by scripts/search_rows.py.

שלב 4

מגדירים איך עונים כשיש או אין מידע

# Answer Rules

## When data exists
Answer in Hebrew.
Use only the provided rows.
If several rows match, summarize the difference between them.

## When data is missing
Say:
"לא מצאתי את המידע הזה בגיליון הנתונים."
Then ask one clarifying question if it can help.

## Forbidden
- Do not say "usually".
- Do not say "probably".
- Do not use external knowledge.
- Do not recommend something unless a matching row supports it.
מילים כמו “בדרך כלל”, “כנראה”, “לרוב” הן נורות אזהרה. הן מסמנות שהמודל מתחיל למלא חורים בעצמו.

שלב 5

מסבירים למודל מה אומרת כל עמודה

זה לא חובה טכנית, אבל זה משפר מאוד את הדיוק. המודל לא צריך לנחש אם `status` הוא סטטוס מוצר, לקוח או משלוח. 

# Sheet Schema

## Columns
- id: stable row identifier.
- category: business area, for example plans, support, billing.
- name: product, plan, policy, or topic name.
- price: numeric monthly price in ILS. Empty means no price listed.
- status: active, archived, draft.
- answer: approved text that may be used in the response.

## Matching guidance
Prefer exact matches in name.
Then use category.
Ignore rows where status is archived unless the user asks about archived items.

שלב 8

כותבים Script שמחפש שורות רלוונטיות

החיפוש הראשוני יכול להיות פשוט מאוד: התאמת מילים בשדות החשובים. בהמשך אפשר להחליף ל־embeddings או חיפוש סמנטי. 

# scripts/search_rows.py
import csv
import json
import sys
from pathlib import Path

DATA_PATH = Path(__file__).resolve().parents[1] / "data" / "products.csv"
SEARCH_COLUMNS = ["category", "name", "answer"]

def normalize(text):
    return (text or "").strip().lower()

def score_row(row, query):
    q = normalize(query)
    score = 0
    for col in SEARCH_COLUMNS:
        value = normalize(row.get(col))
        if q and q in value:
            score += 3
        for token in q.split():
            if len(token) > 1 and token in value:
                score += 1
    if row.get("status") == "archived":
        score -= 2
    return score

def search(query, limit=5):
    with DATA_PATH.open("r", encoding="utf-8-sig", newline="") as f:
        rows = list(csv.DictReader(f))
    ranked = [(score_row(row, query), row) for row in rows]
    matches = [row for score, row in sorted(ranked, reverse=True, key=lambda x: x[0]) if score > 0]
    return matches[:limit]

if __name__ == "__main__":
    query = " ".join(sys.argv[1:]).strip()
    print(json.dumps({"query": query, "rows": search(query)}, ensure_ascii=False, indent=2))

מחברים את ה־LLM רק אחרי שיש Tool יציב

עכשיו האפליקציה הראשית טוענת Markdown, מריצה חיפוש, ושולחת למודל רק את השורות שחזרו. זה הגבול שמחזיק את הסוכן במקום. 

# app.py - pseudo implementation
import json
import subprocess
from pathlib import Path

ROOT = Path(__file__).resolve().parent

def load_markdown():
    files = [
        ROOT / "markdown" / "agent.md",
        ROOT / "markdown" / "answer-rules.md",
        ROOT / "markdown" / "sheet-schema.md",
    ]
    return "\n\n".join(path.read_text(encoding="utf-8") for path in files)

def search_sheet(question):
    result = subprocess.check_output(
        ["python", str(ROOT / "scripts" / "search_rows.py"), question],
        text=True,
        encoding="utf-8"
    )
    return json.loads(result)

def build_messages(question, search_result):
    return [
        {"role": "system", "content": load_markdown()},
        {"role": "user", "content": json.dumps({
            "question": question,
            "sheet_search_result": search_result
        }, ensure_ascii=False)}
    ]

מנסחים הוראת מערכת שמונעת תשובות חופשיות

You are a sheet-grounded answer agent.

You receive:
1. A user question.
2. A JSON object named sheet_search_result.

Answer using only sheet_search_result.rows.
If rows is empty, say in Hebrew that the information was not found in the sheet.
Do not use outside knowledge.
Do not infer missing prices, availability, features, or policy.

Return:
- answer: the user-facing answer in Hebrew
- used_row_ids: row ids used
- confidence: high, medium, or low
- missing_info: what was missing, if anything

שלב 14

אפשר לעבוד גם עם מודל מקומי במקום LLM בענן

באותה ארכיטקטורה בדיוק אפשר להחליף את שכבת ה־LLM: במקום לשלוח את השאלה והשורות ל־API חיצוני, מריצים מודל על המחשב או על שרת פנימי. ה־Markdown, ה־Scripts וה־Sheets נשארים אותו דבר. 

User Question
  -> scripts/search_rows.py
  -> relevant rows as JSON
  -> local model through Ollama / LM Studio / llama.cpp
  -> answer grounded in rows only
מה משתנה?

רק הדרך שבה קוראים למודל. במקום OpenAI/Anthropic/Gemini API, קוראים לשרת מקומי כמו Ollama.

מה לא משתנה?

כללי “ענה רק מהנתונים”, החיפוש ב־Sheet, הלוגים, בדיקות האי־המצאה וה־UI.

זה היתרון הגדול של Markdown Files + Scripts: המודל הוא רכיב להחלפה, לא כל המערכת.

שלב 15

כלים נפוצים להרצת מודלים מקומיים

כלי למי זה מתאים רמת קושי שימוש טיפוסי
Ollama מפתחים שרוצים API מקומי פשוט. נמוכה עד בינונית שרת מקומי ב־localhost, קריאה דרך HTTP.
LM Studio מי שרוצה UI נוח לבחירת מודלים וניסוי. נמוכה בדיקות ידניות, שרת OpenAI-compatible מקומי.
llama.cpp מי שרוצה שליטה עמוקה וביצועים על CPU/GPU. בינונית עד גבוהה הרצה קלה לפריסה, quantization, שרת פנימי.
vLLM / TGI צוותים שמריצים מודלים על שרת GPU. גבוהה פרודקשן, עומס משתמשים, batching, throughput.
להתחלה: Ollama או LM Studio. לפרודקשן כבד: vLLM, TGI, או שירות GPU מנוהל.

שלב 16

דוגמת חיבור מקומי עם Ollama

הסקריפטים עדיין מחזירים JSON מה־Sheet. ההבדל הוא שבמקום לשלוח הודעות לענן, שולחים אותן ל־`http://localhost:11434`. 

# local_llm.py - simplified example
import json
import requests

def ask_local_model(system_prompt, question, sheet_rows):
    payload = {
        "model": "llama3.1:8b",
        "messages": [
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": json.dumps({
                "question": question,
                "sheet_rows": sheet_rows
            }, ensure_ascii=False)}
        ],
        "stream": False
    }
    response = requests.post(
        "http://localhost:11434/api/chat",
        json=payload,
        timeout=120
    )
    response.raise_for_status()
    return response.json()["message"]["content"]
גם במודל מקומי צריך להגביל הקשר: שולחים רק את השורות הרלוונטיות, לא את כל הגיליון אם אין צורך.

שלב 17

השוואה: API בענן מול מודל מקומי

נושא LLM בענן מודל מקומי
קושי התחלה קל: API key, SDK, קריאת HTTP. בינוני: התקנה, הורדת מודל, התאמת חומרה.
איכות תשובות בדרך כלל גבוהה יותר, במיוחד בהוראות מורכבות. תלוי מודל וחומרה. מודלים קטנים עלולים לפספס הוראות.
עלות תשלום לפי שימוש, נוח להתחלה, מצטבר בעומס. אין עלות פר־בקשה, אבל יש עלות מחשב, GPU, חשמל ותחזוקה.
פרטיות מידע נשלח לספק חיצוני לפי תנאי השירות. המידע נשאר אצלך אם כל הריצה מקומית.
תחזוקה הספק מתחזק את המודל והתשתית. אתה אחראי לגרסאות, זיכרון, ביצועים ונפילות.
Latency תלוי רשת וספק, לרוב יציב. יכול להיות מהיר במחשב חזק או איטי מאוד בלי GPU.

שלב 18

ידע נדרש, סביבות פיתוח ועלויות

בענן צריך לדעת

עבודה עם API, ניהול מפתחות, rate limits, מבנה הודעות, JSON mode, tool calling, ולוגים.

במקומי צריך לדעת

מודלים, quantization, זיכרון VRAM/RAM, התקנת runtime, בחירת מודל, ניטור ביצועים.

עלות ענן

נמוכה בהתחלה, עולה עם כמות השאלות ואורך ההקשר. טובה ל־MVP ולצוות קטן.

עלות מקומית

השקעה מוקדמת בחומרה/שרת, תחזוקה וזמן פיתוח. משתלם כשיש עומס גבוה או דרישות פרטיות.

Rule of thumb:
MVP / prototype / best quality quickly -> Cloud LLM
Sensitive data / offline / many calls / full control -> Local model

שלב 19

למה כן ולמה לא בכל שיטה

כן ל־LLM בענן
  • רוצים איכות גבוהה מהר.
  • אין צוות תשתיות או GPU.
  • הדאטה לא רגיש או מותר לשליחה.
  • צריך tool calling ו־JSON אמינים יותר.
לא ל־LLM בענן
  • יש מידע שאסור להוציא החוצה.
  • עלות פר־בקשה עלולה לגדול מדי.
  • צריך עבודה ללא אינטרנט.
  • תלויים מדי בזמינות ספק חיצוני.
כן למודל מקומי
  • פרטיות ושליטה חשובות מאוד.
  • יש הרבה קריאות חוזרות.
  • אפשר להשקיע בחומרה ותחזוקה.
  • רוצים מערכת שעובדת גם offline.
לא למודל מקומי
  • צריך איכות גבוהה בלי התעסקות.
  • אין GPU או שרת מתאים.
  • הצוות לא מכיר inference ו־deployment.
  • המודל המקומי לא מספיק ממושמע להוראות.

  • שדה שאלה ברור.
  • אזור תשובה קצר ונקי.
  • תצוגת “מקורות מהגיליון” עם row id ושם.

  • כפתור “הצג שורות שנקראו” למנהלים.
אל תסתירו את מקורות התשובה. דווקא הצגת השורות מגבירה אמון ומקלה על תיקון הדאטה.

שלב 23

איפה זה בדרך כלל נשבר

בעיה סימן תיקון
החיפוש מחזיר שורות לא קשורות התשובה נכונה תחבירית אבל מוזרה. לשפר scoring, להוסיף category, או embeddings.
המודל ממציא יש מידע שלא קיים בשורות. להקשיח Markdown, להחזיר JSON, לבדוק `used_row_ids`.
הטבלה לא עקבית אותו מוצר כתוב בכמה שמות. להוסיף id, aliases, validation.
אין אמון בתשובה המשתמש לא יודע מאיפה המידע. להציג מקורות מהגיליון ליד התשובה.

תבנית העבודה

הסוכן הוא מערכת קטנה, לא פרומפט ארוך

Markdown = זהות, כללים, תהליך, גבולות
Scripts  = קריאה, חיפוש, בדיקה, JSON
Sheets   = מקור האמת העסקי
LLM      = הבנה, תיאום, ניסוח
Model    = Cloud API או Local runtime, לפי פרטיות, עלות ואיכות
UI       = אמון, מקורות, מצב ריק, בקרה
הדרך הנכונה להתחיל: CSV מקומי, חיפוש פשוט, Markdown קשיח, תשובות JSON. רק אחרי שזה יציב בוחרים אם לרוץ עם Cloud LLM או מודל מקומי, ואז עוברים ל־Google Sheets API, embeddings והרשאות מתקדמות.

אם שומרים על העיקרון “עונים רק ממה שהסקריפט החזיר”, מקבלים סוכן פשוט, שימושי, ואמין הרבה יותר מסוכן שמנסה לדעת הכול לבד.
